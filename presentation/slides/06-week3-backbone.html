Week 3 · Transformer architecture

Ingestion backbone + dynamic few-shot retrieval

Two non-LLM layers before the model: the ingestion + topic-filter backbone, and a dynamic few-shot retriever that balances examples across category and label — so the model learns the structure of a claim, not a topic shortcut. Baseline set at recall 0.792, with the null hypothesis openly on the table.

[2:10–2:40] — ~30s. My densest week; keep it moving.

In Week 3 I built two layers, both with no LLM in the path.

Ingestion: keyword search over Bluesky and GDELT, designed to run every 24 hours. Then a pure-Python topic filter keeps only North-American extreme-climate content. Cheap, deterministic, and it keeps junk away from the expensive model.

The real ML work is the dynamic few-shot retriever. Instead of one fixed set of examples, I embed labeled posts into ChromaDB with all-MiniLM and, per query, pull the top-2 from every category × label bucket — 20 examples, always label-balanced. The reason: pure similarity retrieval takes a topic shortcut — a wildfire query returns ten wildfire examples and the model learns "wildfire equals claim" instead of the structural pattern. Stratifying forces it to see the claim-versus-opinion distinction across all five domains. Sparse categories get a two-pass backfill so label balance is never broken. And the eval posts are excluded from the store — no leakage.

Evaluating this is what drove the model switch: gemma2:2b was slow and weak on recall, so I moved to qwen2.5:3b, which set the baseline — recall 0.792 on the 100-post eval, fixed 8-shot. The open question, stated as a null hypothesis: can stratified retrieval beat eight well-chosen fixed examples, especially on the hard conspiracy and sensationalist categories? Week 4 answers it.
