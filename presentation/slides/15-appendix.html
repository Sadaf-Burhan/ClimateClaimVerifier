Appendix · How a red flag is raised

Q: "How do you avoid false accusations?" — this slide.

Eight conditions, all of which must hold. Two raise suspicion; six can defuse it. Every guard was earned from a real false flag in real data.

The best one: "I saw them spraying chemtrails over my neighborhood this morning" reads as a first-person local eyewitness report, so the eyewitness guard would happily defuse it. A conspiracy check runs first on both defusers and refuses to let them fire. Casualness is a trap.

The citation matcher is exact-or-dotted-suffix, so evil-nature.com cannot impersonate nature.com. Same rule protects the official allowlist.

Known gap: roughly 67 posts in the corpus cite through URL shorteners, which hide the real domain and get no credit — so they can draw a false "no cited evidence" flag.

Appendix · How it actually runs

Q: "Is this actually deployed / how does it scale?"

Two loops that never call each other — they meet only through files on disk.

The reason is a direct consequence of Week 4. Single-post inference is what hits the recall target, and it costs ~16x the LLM calls. So ingestion is cheap and runs on a laptop; the LLM backlog is batched onto a GPU. The classifier's recall win dictated the deployment topology.

Be honest if asked: a single unified run_daily() entrypoint is designed but not built — today it's two commands. The chaining exists; the one-button version doesn't.

Appendix · Evaluation discipline

Q: "How do you know it isn't silently degrading?"

Every eval run is stamped with the model digest and the Ollama version, and appended to a drift log. That's how I know the 0.9583 → 0.875 move is real and not a runtime change.

There are two eval sets on purpose: a frozen gold set of 100 rows that is never relabelled, and a dynamic set that grows as I adjudicate new edge cases — currently 112 rows. Gold catches regression; dynamic catches concept drift.

Honest caveat: they overlap heavily right now (~99 shared rows), so the dynamic set can barely register drift yet. It needs to diverge before it earns its name.

The bar was recall ≥ 0.90 AND precision ≥ 0.85. The best adapter beat the base on precision by 0.04 — while giving up 0.08 recall. Not a trade worth making when a downstream RAG layer recovers precision for free.

Q: "Did you try fine-tuning?" / "Why not just train it more?"

These are the Week-5 measurements, taken at the time on the then-current eval set — that's why the base row reads 0.938 rather than the 0.875 I show on the results slide. The relative comparison is what matters here: all four rows were measured under identical conditions.

The frontier is real and it's a data-volume and model-size limit, not label noise. ~120 training examples against a 3B model.

If pushed on what would move it: more labelled data, or a bigger base. Both were out of scope for a local-only, zero-cost build.

Appendix · Known gaps

What I'd fix, in order

  • No retrieval eval. The precision workhorse is the one stage with no number.
  • Recall sits at 0.875, below the 0.90 bar — six specific rows, the same six every run.
  • Headline-only corpus. GDELT returns titles, not article bodies. A full news parser was ruled out of scope — bot walls, paywalls, JS rendering.
  • URL shorteners hide citations. ~67 posts cite through bit.ly and friends and get no credit for it.
  • run_daily() isn't built. The autonomous loop is designed and chained across two commands, not one.
  • The dynamic eval set barely diverges from gold (~99 shared rows), so it can't detect concept drift yet.

If asked "what's missing" — this is the list, ordered by how much it would actually change my confidence in the system.

The framing to hold: "no match" is ambiguous, not negative. Everything the system says is scoped to the retrieved set. It never claims no evidence exists in the world — only that it found none in what it searched. That's the honest limit of a headline-only corpus, and it's why the UI wording says "in our set."
